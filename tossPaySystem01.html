대상: myPG01 (기존) vs toss-payment-admin (현재 프로젝트)

1. 기술 스택 및 인프라

항목 myPG01 toss-payment-admin
디렉터리 구조 src/app/ app/ (App Router 직접)
CSS 방식 순수 CSS (Variables) Tailwind CSS
HTTP 클라이언트 Axios Native fetch
DB SQLite (파일 기반) PostgreSQL
다크 모드 ✅ 지원 ❌ 미지원

2. 토스페이먼츠 SDK 비교

항목 myPG01 toss-payment-admin
SDK 버전 payment-widget-sdk (구) tosspayments-sdk (신)
결제 UI 위젯 임베드 (페이지 내) 리다이렉트 (외부 이동)
약관 동의 위젯 내 자동 포함 별도 처리 없음
UX 경험 페이지 이탈 없음 외부 결제창 이동 후 복귀

3. 인증 및 데이터 모델링

항목 myPG01 toss-payment-admin
로그인 식별 username email
비밀번호 보안 평문 저장 (위험) 해시 저장 (bcrypt)
사용자 연결 ❌ 없음 ✅ FK 연동
결제 상태 3단계 (단순) 7단계 (상세)

4. 어드민 및 운영 기능

항목 myPG01 toss-payment-admin
UI 구조 단일 페이지 대시보드 + 상세 페이지
KPI 통계 ❌ 없음 ✅ 매출/취소 통계
환불 방식 전액 취소만 가능 금액 지정 부분 취소
목록 편의성 단순 나열 검색/필터/페이징

💡 종합 평가 및 제안

현재 상황 요약:
  • myPG01: 프로토타입 단계이나, 위젯 방식의 UX와 다크모드 대응이 장점임.
  • toss-payment-admin: 보안, 확장성, 운영 편의성을 갖춘 실무 최적화 시스템임.

향후 개선 방향: toss-payment-admin의 견고한 설계 위에 myPG01의 임베드형 결제 위젯 로직을 결합하여 결제 전환율과 보안성을 동시에 확보할 것을 권장함.
